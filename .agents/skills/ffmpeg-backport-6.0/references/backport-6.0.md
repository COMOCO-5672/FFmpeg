# Backport 6.0 Reference

## 1. 回移植时先问清楚的问题

- 这个 patch 修的是正确性、稳定性还是性能
- 主线 patch 依赖了哪些 6.0 没有的基础设施
- 是否可以只迁最小必要逻辑
- 是否会破坏本地已有定制

## 2. 常见差异点

- 结构体字段不同
- helper API 名称不同
- codec 注册和构建开关位置不同
- log 或 option 框架细节不同
- fftools 层参数处理有差异

## 3. 实施顺序

### 3.1 先保行为

先把功能跑起来，不要第一步就追求代码外观完全一致。

### 3.2 再保编译

处理 `implicit declaration`、字段缺失、宏差异、类型不匹配。

### 3.3 再保回归

至少验证：

- 新样本已修复
- 旧样本不回归
- Windows/MSYS2 本地构建通过

## 4. 提交拆分模板

- commit 1: backport 核心功能
- commit 2: 6.0 兼容适配
- commit 3: 注释、日志、清理

这样做便于后续 bisect 和继续回移植。
